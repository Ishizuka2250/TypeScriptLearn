JavaScript vs TypeScript

Day3 導入ガイド

TypeScript が JavaScript に何を加えるのか、コードで理解する。

1. TypeScript とは

TypeScript は JavaScript のスーパーセットです。 すべての JavaScript コードはそのまま TypeScript として動作します。 TypeScript が加えるのは主に 静的型システム です。

.ts ファイル
→
tsc（コンパイラ）
→
.js ファイル
→
Node.js / ブラウザ

実行されるのは変換後の JavaScript です。型情報はコンパイル時にのみ存在し、実行時には消えます。

💡
PHP と比べると: PHP も型宣言を書かなくても動きます（動的型）。TypeScript は PHP の strict_types=1 を強制した状態に近いイメージです。ただし TypeScript の型はあくまでコンパイル時の検査で、実行時の保証ではありません。

2. JavaScript との主な違い

項目 観点 JavaScript TypeScript
型付け いつ検査するか 動的（実行時） 静的（コンパイル時）
型注釈 書き方 なし 変数: 型 で明示
インターフェース オブジェクト構造の定義 なし（JSDoc のみ） interface / type
エラー検出 タイミング 実行して初めてわかる エディタ上でリアルタイム
ファイル拡張子 — .js / .mjs .ts / .tsx（JSX含む）
実行方法 — そのまま Node.js で実行 コンパイルしてから実行（または ts-node）

制御構文は JavaScript と完全に同じ

if / for / while / async await などの制御構文は JavaScript と一切変わりません。TypeScript は JavaScript を拡張したものであり、既存の構文はそのまま使えます。

制御構文は JS と同じ — 型注釈だけが TypeScript 固有
// 制御構文: JS と全く同じ
for (const item of items) {
  if (item.qty < 10) {
    console.log(item.name);
  }
}

// TypeScript 固有: 型注釈（: の右側）
const qty: number = 10;            // ← : number が TS 固有
interface Item { name: string; }  // ← interface 宣言が TS 固有

TypeScript で新しく覚えることは3つだけ

追加される構文 例 用途
型注釈 qty: number 変数・引数・戻り値に型を付ける
型定義 interface Item { ... } / type MovementType = "in" | "out" オブジェクト構造やユニオン型を定義する
ジェネリクス Array<Item> / Promise<Stock> 型を引数として受け取る汎用的な型
✅
「JavaScript が書ければ TypeScript も8割は書ける」と言われる理由はここにあります。Day3 では残り2割（型の書き方）を集中して習得します。

3. 型注釈の基本構文

変数・関数の引数・戻り値

JavaScript
// 型の記述なし
let qty = 10;
let name = "ペン";

function calcTotal(price, count) {
  return price * count;
}

// 誤った使い方でも実行まで気づかない
calcTotal("100", 5); // "1005050..." になる
TypeScript
// : 型 で注釈を付ける
let qty: number = 10;
let name: string = "ペン";

function calcTotal(
  price: number,
  count: number
): number {
  return price * count;
}

// コンパイルエラー（エディタで赤線）
calcTotal("100", 5); // Error: string は number に渡せない

プリミティブ型の一覧

string number boolean null undefined any — 型チェック無効（使用は最小限に） unknown — any より安全な型不明 void — 戻り値なし関数 never — 到達しないパス

4. interface と type — オブジェクトの型定義

📦
在庫管理で扱う 商品・在庫・入出庫 などのデータ構造を TypeScript の型として表現します。PHP の class に型注釈を付けたものに近いイメージです。
interface — 拡張しやすい
interface Item {
  id: number;
  name: string;
  category: string;
  price: number;
}

// extends で拡張できる
interface ItemWithStock extends Item {
  qty: number;
}

const pen: ItemWithStock = {
  id: 1,
  name: "ペン",
  category: "文具",
  price: 120,
  qty: 50,
};
type — ユニオン型など柔軟な定義
type MovementType = "in" | "out";
// "in" か "out" の文字列のみ許可

type StockMovement = {
  sku: string;
  type: MovementType;
  qty: number;
};

const m: StockMovement = {
  sku: "A",
  type: "in",
  qty: 100,
};

// コンパイルエラー
m.type = "invalid"; // Error!
✅
interface vs type どちらを使う？
オブジェクト型の定義には interface、ユニオン型（"in" | "out"）や型の組み合わせには type を使うのが一般的なルールです。迷ったら interface から始めて問題ありません。

5. よくある型エラー パターン

パターン1: null / undefined のアクセス

JavaScript（実行時エラー）
const item = findItem(99); // 存在しない
// item = undefined

console.log(item.name);
// 実行してみて初めてクラッシュ
// TypeError: Cannot read properties
// of undefined (reading 'name')
TypeScript（コンパイル時に検出）
function findItem(id: number): Item | undefined {
  return items.find(i => i.id === id);
}

const item = findItem(99);

console.log(item.name);
// Error: 'item' is possibly 'undefined'

// 修正: 存在チェックを強制される
if (item) console.log(item.name); // OK

パターン2: 配列の要素型

JavaScript
const skus = ["A", "B", "C"];

// 誤った型を入れても気づかない
skus.push(123);    // 動いてしまう
skus.push(true);   // 動いてしまう
TypeScript
const skus: string[] = ["A", "B", "C"];

skus.push(123);   // Error: number は string[] に追加不可
skus.push(true);  // Error: boolean は string[] に追加不可

skus.push("D");   // OK

パターン3: 関数の戻り値型

JavaScript
function getStock(sku) {
  // 返し忘れても気づかない
  if (sku === "A") {
    return { sku: "A", qty: 10 };
  }
  // else のとき undefined を返す
}
TypeScript
type Stock = { sku: string; qty: number };

function getStock(sku: string): Stock {
  if (sku === "A") {
    return { sku: "A", qty: 10 };
  }
  // Error: Stock を返す必要があるが
  // undefined を返しているパスがある
}

6. strict モードとは

tsconfig.json に "strict": true を書くと、以下のチェックがすべて有効になります。

✓ strictNullChecks — null/undefined の見落としを防ぐ
✓ noImplicitAny — 型推論できない変数を any にしない
✓ strictFunctionTypes — 関数型の厳密なチェック
✓ strictPropertyInitialization — クラスのプロパティ初期化を強制
tsconfig.json（Day3 で使う最小構成）
{
  "compilerOptions": {
    "target": "ES2020",       // 出力する JS のバージョン
    "module": "commonjs",    // Node.js 向けモジュール形式
    "strict": true,          // 厳密モード（全チェック有効）
    "outDir": "./dist",      // コンパイル後 JS の出力先
    "rootDir": "./src"       // ソースファイルのルート
  }
}
⚠️
any 型は TypeScript のすべての型チェックを無効にします。「とりあえずコンパイルを通す」ために使いたくなりますが、乱用すると TypeScript を使う意味がなくなります。型がわからないときは unknown を使い、必要に応じてナローイング（型を絞り込む処理）してください。
