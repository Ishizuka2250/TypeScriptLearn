# TypeScript / Next.js 学習カリキュラム（1〜2か月）

## 前提
- 対象案件: 小規模在庫管理システム
- 現在地: Laravel中心の実務経験あり、JavaScript実務経験あり
- 目標: TypeScript + Next.js で業務アプリを一通り構築できる状態

## 全体設計
- 期間A（短期1か月）: 実装優先の最短ルート（4週間）
- 期間B（標準2か月）: 設計・品質まで含めた実務ルート（8週間）
- 学習単位: 週ごとのゴール + 実装課題 + 完了条件

## 期間A: 1か月（4週間）

### Week 1: TypeScript基礎を業務目線で固める
- テーマ
  - 型注釈、interface/type、Union、Generics、型ガード、strictモード
- 課題
  - 在庫アイテム型、入出庫トランザクション型、APIレスポンス型を定義
  - バリデーション関数（型ガード）を実装
- 完了条件
  - anyを使わずに主要ドメイン型を定義できる
  - 不正データを型ガードで弾ける

### Week 2: Next.js App Router + 画面構築
- テーマ
  - App Router、レイアウト、Server/Client Components、フォーム入力
- 課題
  - 在庫一覧、商品登録、商品詳細の3画面を作成
  - 画面遷移と入力バリデーションを実装
- 完了条件
  - 主要画面が動作し、型エラーなしでビルド可能

### Week 3: API / DB連携（Prisma + SQLite or MySQL）
- テーマ
  - Route Handlers、Prismaスキーマ、CRUD、エラーハンドリング
- 課題
  - 商品マスタCRUD、在庫増減API、検索APIを実装
  - 在庫数がマイナスにならない制約を実装
- 完了条件
  - API経由で在庫更新ができる
  - 主要異常系をハンドリングできる

### Week 4: 最小品質担保 + リリース準備
- テーマ
  - 単体テスト（Vitest/Jest）、E2E（Playwright）、環境変数管理
- 課題
  - 重要ロジック（在庫加減算、バリデーション）をテスト
  - READMEにセットアップ手順・設計方針・制約事項を記載
- 完了条件
  - テスト通過、ローカルで再現可能、説明可能

## 期間B: 2か月（8週間）

### Week 1-2: TypeScript強化
- 型設計演習（DTO、Domain Model、Repository層の型）
- Zodで入力バリデーションを型と同期
- 課題: API契約を型で表現し、フロント/バックで共有

### Week 3-4: Next.js実践
- App Router深掘り（Server Actions、データフェッチ戦略）
- 認証（NextAuth.jsまたはAuth.js）導入
- 課題: ログイン必須の在庫更新フローを実装

### Week 5-6: DB設計と業務ロジック
- 在庫管理向けER設計（items, stocks, stock_movements, users）
- トランザクション・整合性・楽観ロックの考え方
- 課題: 入庫/出庫履歴と在庫一覧の整合を保つ

### Week 7: テスト/品質
- Unit/Integration/E2Eの役割分担
- ESLint + Prettier + 型チェックをCIで自動化
- 課題: PR時にlint/test/typecheckを必須化

### Week 8: ミニ本番想定
- デプロイ（Vercel想定）
- 監視の入口（ログ確認、エラーハンドリング方針）
- 課題: 「要件定義→実装→テスト→デプロイ」一連を実施

## 推奨ミニプロダクト要件（学習課題共通）
- 商品マスタ管理（作成・更新・一覧・詳細）
- 在庫入出庫登録（理由、数量、担当者、日時）
- 在庫一覧（閾値以下を警告表示）
- 検索/絞り込み（商品名、カテゴリ、在庫状態）
- 操作ログの表示（最低限）

## 週次レビュー項目
- 何を作ったか（動作するURL/画面）
- 詰まった点と原因
- 次週改善する設計ポイント
- 型安全・テスト・UIの自己採点（5段階）

## 学習リズム（目安）
- 平日
  - 30分: インプット
  - 60〜90分: 実装
  - 10分: 振り返りメモ
- 休日
  - 2〜3時間: 課題実装
  - 30分: テスト追加/リファクタ

## 到達判定（案件投入OKの目安）
- TypeScript strictで主要機能を実装できる
- Next.js App RouterでCRUD + 認証 + DB連携ができる
- 最低限のテストとエラーハンドリングを説明できる
- 在庫管理の業務シナリオを自分で分解して実装できる
